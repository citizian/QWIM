# QWIM 即时通讯服务端通信协议及接口说明文档

本文档是给客户端开发小组的参考指南，详细描述了QWIM即时通讯客户端与服务端的交互协议、数据包格式以及各业务接口说明。

## 1. 通信基础
- **传输层协议**: TCP
- **默认端口**: `8081` (可由服务端配置文件指定，客户端开发期间请确认实际环境端口)
- **长连接与心跳机制**:
  - 客户端连接后需在指定时间内发送任意数据包（默认心跳超时时间为 **30秒**），否则服务端将强制断开连接。
  - 为维持连接，客户端应定期（如每隔15秒）发送 `heartbeat` 类型的消息以重置服务端的超时计时器。

## 2. 数据包格式 (Framing)
TCP是流式协议，为了解决粘包和半包问题，服务端采用了 **显式长度前缀 (Length-Prefixed)** 的帧格式封装数据。

每条完整的消息由两部分组成：`包头(Header)` + `载荷(Payload)`
- **Header**: 4字节，表示负载长度 `N`。**大端序 (Network Byte Order, Big-Endian)**。
- **Payload**: 长度为 `N` 字节的字符串，内容为UTF-8编码的 **JSON** 格式字符串。

**示意图**:
```text
+-----------------------+----------------------------------+
| Msg Length (4 bytes)  | JSON Payload (N bytes)           |
| (Big Endian uint32_t) | {"type": "...", ...}             |
+-----------------------+----------------------------------+
```

## 3. 业务接口说明 (JSON Payload)
所有有效业务的数据正文均为一层JSON对象，必需包含 `type` 字段，服务端通过 `type` 进行路由解析。

### 3.1 用户业务

#### 登录 (Login)
- **场景**: 建立TCP连接后调用的首个业务包，用于向服务端注册用户名。
- **请求 (Client -> Server)**:
  ```json
  {
      "type": "login",
      "user": "你的用户名"
  }
  ```
- **服务端响应**:
  1. 向**当前客户端**逐条推送近期历史消息（最多50条）：
  ```json
  {
      "type": "history",
      "msg": "发送者名: 消息内容"
  }
  ```
  2. 向**其他所有在线用户**广播上线通知：
  ```json
  {
      "type": "system",
      "msg": "你的用户名 joined"
  }
  ```

#### 获取在线列表 (List)
- **场景**: 主动拉取当前所有在线用户的用户名。
- **请求 (Client -> Server)**:
  ```json
  {
      "type": "list"
  }
  ```
- **服务端响应 (Server -> Client)**:
  ```json
  {
      "type": "list",
      "users": ["user1", "user2", "你的用户名"]
  }
  ```

### 3.2 消息业务

#### 发送群聊消息 (Chat)
- **场景**: 在大厅发布消息，所有人都能收到。
- **请求 (Client -> Server)**:
  ```json
  {
      "type": "chat",
      "msg": "你好，所有人！"
  }
  ```
- **服务端响应**:
  - 群发给 **除发送者外** 的所有在线用户：
  ```json
  {
      "type": "chat",
      "user": "发送者用户名",
      "msg": "你好，所有人！"
  }
  ```

#### 发送私聊消息 (Private)
- **场景**: 向指定用户单独发送消息。
- **请求 (Client -> Server)**:
  ```json
  {
      "type": "private",
      "to": "目标用户名",
      "msg": "你好，这是一条私信"
  }
  ```
- **服务端响应**:
  1. 如果目标在线，**目标用户**将收到：
  ```json
  {
      "type": "private",
      "user": "发送者用户名",
      "msg": "你好，这是一条私信"
  }
  ```
  2. 如果目标不在线，**发送者本人**将收到报错提醒：
  ```json
  {
      "type": "system",
      "msg": "user not online"
  }
  ```

### 3.3 系统与维持业务

#### 心跳保活 (Heartbeat)
- **场景**: 定时发送以维持TCP连接有效性。
- **请求 (Client -> Server)**:
  ```json
  {
      "type": "heartbeat"
  }
  ```
- **服务端响应**: `无` (仅在服务端刷新计时器)。

#### 系统消息 (System)
- **场景**: 服务端单向推送的通知类消息，客户端需要监听该 `type` 并做UI渲染。
- **格式 (Server -> Client)**:
  ```json
  {
      "type": "system",
      "msg": "xxx left" // 其他可能的值包括 "xxx joined" 或 "user not online"
  }
  ```

## 4. 客户端开发建议
1. **TCP拆包与粘包处理**：接收数据时不能假设一次recv就是一个完整的包。需先从缓冲器读取4个字节解析出包长（需调用 `ntohl` 或等效接口转回主机字节序），当缓冲区中剩余可读字节数 `>= 包长` 时，再完整切分并交由JSON解析器。
2. **重连机制**：若因网络抖动或心跳超时导致被断开，Socket重连后需重新发送 `login` 报文恢复会话，否则服务端无法识别新连接的用户身份。
3. **自身消息展示**：服务端在广播 `chat` 大厅消息时**默认不会回传**给发送者本人。客户端在成功向套接字写入群聊包后，建议直接在本地UI做消息上屏。

## 5. 迭代更新日志 (Changelog)

### v1.1.0 - 核心机制与数据持久化迭代 (2026-04-03)
本次更新增强了身份核心鉴权且开启了所有聊天系统的全量数据库落盘。

#### 5.1 新增：账号注册 (Register)
- **请求入参 (Client -> Server)**:
  ```json
  {
      "type": "register",
      "user": "新用户名",
      "password": "密码明文"
  }
  ```
- **服务端响应 (Server -> Client)**:
  ```json
  {
      "type": "system",
      "msg": "register success" // 若已经被注册则会返回 register failed:...
  }
  ```

#### 5.2 迭代：登录接口强化鉴权与免密重连 (Login)
- **变更一：** 经典的 `login` 路由必须附注明文 `password` 进行对比：
  ```json
  {
      "type": "login",
      "user": "用户名",
      "password": "你的密码"
  }
  ```

- **变更二（新增响应）：** 当客户端初次通过密码验证后，会在获取 `history` 服务器历史推送前，率先收到一条下发 Token 的独有通讯包：
  ```json
  {
      "type": "login_success",
      "token": "eyJhbGciOiJIUzI... (属于该用户的独有持久化凭证)"
  }
  ```

- **变更三（快速重连机制）：** 客户端在发生断线情况重新建立 TCP 握手时，可以将上面存盘的 `token` 下发给服务器，服务器支持纯令牌直接秒连。
  ```json
  {
      "type": "login",
      "token": "之前拿到的持久化凭证字符串"
  }
  ```

#### 5.3 其他特性强化
- **无感安全下发**：所有报文的持久化在进入 MariaDB 时均已做了防 SQL 注入与单向 Hash 脱敏防御（无需由客户端代工）。
- **漫游记录**：所有服务端的聊天数据全量从易失性内存转移到了数据库内。这意味着任何用户无论何时上线，或者服务端发生故障重启，世界频道的消息环境都不会清零了。
