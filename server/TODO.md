## Phase 2: IM 通讯场景深化升级 (Medium Priority)
- [ ] **a. 离线消息管理系统**：如果发送 `private` 消息时目标不在线，记录入库并在其下次发出 `login` 请求时推送离线期间的缓存消息。
- [ ] **b. 群组与多房间机制**：废弃纯粹无差别的全局广播，增加具体概念上的室（`room_id`）和组（`group`）。支持建群、进群、退群、按群发送路由消息。
- [ ] **c. 消息链路可靠性 (ACK)**：服务端入库消息落盘后，对客户端进行收执应答（ACK），并在协议段内补充全局唯一 ID 以防止消息被漏收或重发导致重复。
- [ ] **d. 支持多端多图文**：设计支持富媒体文件、图片的协议字段拓展规划；如果含有附件，提供对象存储（OSS/S3 等）相关上传凭证接口的设计。

## Phase 3: 高可用与分布式演进 (Low Priority)
- [ ] **a. 心跳包状态同步**：优化 `heartbeat` 超时被动断联逻辑，结合客户端探活时间以及 Redis TTL 维护更加稳定精准的连接态监测。
- [ ] **b. 接入反向代理负载均衡**：配合 Nginx / LVS 等环境组件测试分布式请求分发。
- [ ] **c. 进程间（跨服）路由**：使用类似于 Redis Pub/Sub（发布订阅）或者消息队列中间件 Kafka / RabbitMQ 管理分布式实例之间横向的数据连通和收发路由通信。
